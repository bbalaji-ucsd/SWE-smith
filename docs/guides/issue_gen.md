# Generate Issue Text

You have a bunch of task instances with executable environments.
You're very close to training SWE-agents on this data.
There's one last step - let's generate issue text.

We primarily use LMs to generate issue text.

```bash
python -m swesmith.issue_gen.generate \
    -d logs/task_insts/<repo>.json \
    -c configs/issue_gen/ig_v2.yaml \
    -w 1
```

The model is specified inside the config YAML file (the `model` field). The default config `ig_v2.yaml` uses `anthropic/claude-sonnet-4-20250514`.

This will generate issue text for each task instance, producing several artifacts along the way:

* Under `logs/issue_gen/<repo>`, there will be a JSON file for each task instance containing:
    * `messages`: The messages fed to the LM to generate the issue text.
    * `responses`: The generated issue text, keyed by model name.
    * `cost`: Inference cost.
* A `logs/task_insts/<repo>__ig_llm.json` file will be created, which is a copy of the original file with issue text added to each task instance (as the `problem_statement` field).

### CLI Options

| Flag | Description |
|------|-------------|
| `-d`, `--dataset_path` | Path to the task instances JSON (output of `harness.gather`) |
| `-c`, `--config_file` | Path to the issue generation config YAML |
| `-w`, `--workers` | Number of parallel workers (default: 1) |
| `-i`, `--instance_ids` | Filter to specific instance IDs (supports glob patterns) |
| `-r`, `--redo_existing` | Regenerate issue text for instances that already have output |
| `--user` | GitHub personal account to clone from (mutually exclusive with `--org`) |
| `--org` | GitHub organization to clone from (mutually exclusive with `--user`) |

### Using AWS Bedrock

If you have access to AWS Bedrock instead of direct API keys, create a copy of the config with a Bedrock model string:

```yaml
# configs/issue_gen/ig_v2_bedrock.yaml
model: bedrock/us.anthropic.claude-sonnet-4-6
```

Then run with the Bedrock config:

```bash
python -m swesmith.issue_gen.generate \
    -d logs/task_insts/<repo>.json \
    -c configs/issue_gen/ig_v2_bedrock.yaml \
    -w 1
```

See the [Using AWS Bedrock](../guides/bedrock.md) guide for setup details.

## Alternatives

In our paper, we discuss several alternatives for generating issue text.
While our experiments suggest that LM generated issue text is the best proxy for real issue text, we provide instructions for the alternatives below.

**Static Issue Text**

The problem statement is generated by randomly selecting one of 7 static issue text templates.

```bash
python -m swesmith.issue_gen.get_static logs/task_insts/<repo>.json
```

Produces a `logs/issue_gen/<repo>__ig_static.json` file.

**Random F2P Test Case**

The problem statement shows a randomly selected Fail-to-Pass test case from the task instance.

```bash
python -m swesmith.issue_gen.get_from_tests logs/task_insts/<repo>.json
```

**Original Issue Text**

!!! note
    This strategy only works for some PR Mirrors, if the pull request the mirror is based on has issue(s) associated with it.

```bash
python -m swesmith.issue_gen.get_from_pr logs/task_insts/<repo>.json
```

Produces a `logs/issue_gen/<repo>__ig_orig.json` file.
